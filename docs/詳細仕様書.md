## 1. システム概要

### 1.1 目的

複数店舗の **棚卸し入力・補充依頼・移動記録・在庫参照** を、スマートフォン／PC ブラウザから一元操作し、**Google スプレッドシート**（GAS バックエンド）と連携する。

### 1.2 アーキテクチャ

```
[ブラウザ: Vue 3 SPA（Cloudflare Pages 等で静的ホスティング）]
        │ HTTPS (fetch)
        ▼
[GAS Web アプリ URL（/exec）]
        │
        ▼
[Google スプレッドシート（1ファイル・月別シート構成）]
```

- フロントは **静的ホスティング**（Cloudflare Pages 想定）
- 認証は **PIN（4桁）** をクライアント側で検証（GAS 側でも `apiKey` パラメータを検証）
- API キー・GAS URL は **環境変数**（`VITE_*`）で注入

### 1.3 技術スタック

| 区分 | 技術 |
|------|------|
| フレームワーク | Vue 3（Options API） |
| ビルド | Vite 5 |
| スタイル | Tailwind CSS 3 |
| HTTP | `fetch`（`src/api.js` に集約） |
| ホスティング | Cloudflare Pages（または同等の静的配信） |
| バックエンド | Google Apps Script（`Code.gs`） |
| データストア | Google スプレッドシート（1ファイル・月別シート） |

## 4. アプリケーション状態モデル（`App.vue`）

### 4.1 `appMode`

| 値 | 意味 |
|----|------|
| `null` | ポータル |
| `'inventory'` | 棚卸し入力 |
| `'request'` | 補充依頼 |
| `'transfer'` | 移動記録 |
| `'stock'` | 在庫量確認 |

### 4.2 ステップ管理

| 変数 | 型 | 用途 |
|------|----|------|
| `currentStep` | `Number` | 棚卸し・補充依頼のステップ（0 = トップ） |
| `transferStep` | `Number \| String` | 移動記録のステップ（`0`, `'1a'`, `'2a'`, `'1b'`） |

### 4.3 ブラウザ履歴

`history.pushState` / `replaceState` で `{ appMode, currentStep, transferStep }` を保持する。`popstate` イベントをインターセプトし、現在状態を再 push した上でフッター戻るロジック（`prevStep` / `returnToPortal`）を呼ぶことで、**ブラウザバックと戻るボタンが常に同一の挙動**になる。

### 4.4 グローバル確認ダイアログ（`ConfirmDialog.vue`）

- `App.vue` が `provide()` で `requestConfirm(message, okLabel?, okClass?)` を提供する（Promise を返す）
- 子コンポーネントは `inject: ['requestConfirm']` で利用
- ネイティブ `window.confirm()` は一切使用しない（モバイル/PWA 環境でのブロック対策）

### 4.5 グローバル UI

- **ローディング**: `LoadingOverlay`（子から `loading` / `loadingMessage` を emit）
- **ブランド絞り込み**: `BrandFilterSheet`（モードごとに `brands` ソースを切替）

## 5. バックエンド API（`api.js` / `Code.gs`）

### 5.1 通信方式

| メソッド | 関数 | 備考 |
|----------|------|------|
| GET | `gasGet(action, params)` | クエリ: `action`, `apiKey`, その他パラメータ |
| POST | `gasPost(action, payload)` | Body は **`Content-Type: text/plain`**（GAS CORS プリフライト回避）、JSON 文字列: `{ action, apiKey, payload }` |

エラー時: レスポンス JSON の `error.message` を `throw`（GET/POST 共通）。

### 5.2 アクション一覧

| フロント関数 | GAS `action` | 主な用途 |
|---|---|---|
| `getSheetData(storeKey, sheetName)` | `getSheetData` | **棚卸し** 行データ取得（`sheetName` は月ラベル: `4月` 等） |
| `submitData(payload)` | `submitData` | **棚卸し** 送信・シート書き込み |
| `checkNegativeConsumption(payload)` | `checkNegativeConsumption` | **棚卸し** 前月消費量マイナスのドライラン検証 |
| `getInventoryData(monthNum)` | `getInventoryData` | **補充依頼** 在庫一覧取得（数値月） |
| `getFlavorListForTransfer(monthNum)` | `getFlavorListForTransfer` | **移動記録** フレーバー一覧（`stock` 含む） |
| `getPendingTransferRecords(monthNum, destStoreKey)` | `getPendingTransferRecords` | **移動記録** 未検品一覧取得 |
| `getTransferRecordDetail(monthNum, blockIndex, destStoreKey)` | `getTransferRecordDetail` | **移動記録** 検品詳細取得 |
| `submitTransferRecord(payload)` | `submitTransferRecord` | **移動記録** 起票・入荷・廃棄の書き込み |
| `addFlavorForArrival(payload)` | `addFlavorForArrival` | **移動記録(入荷)** 新規銘柄行の追加（A/B列） |
| `completeInspection(payload)` | `completeInspection` | **移動記録** 検品完了・数量上書き |
| `getStockOverview(monthNum)` | `getStockOverview` | **在庫量確認** 一覧取得 |

> **重要（棚卸し）:** データ取得は必ず `getSheetData` を使用する。`getInventoryData` は補充依頼専用であり、行番号（`rowIndex`）の体系が異なるため、誤用すると GAS 側でシート書き込みエラーが発生する。

### 5.3 スプレッドシート列定義（`Code.gs`）

#### 5.3.1 店舗キー対応

| キー | 店舗名 | 棚卸し列群 | 補充依頼キー |
|------|--------|------------|-------------|
| `office` | 事務所 | G〜Y | `office` |
| `baba` | 馬場本店 | AB〜AT | `baba_main` |
| `nakano` | 中野店 | AW〜BO | `nakano` |
| `baba_2nd` | 馬場2号店 | BR〜CJ | `baba_2nd` |

> **注意:** 棚卸しと補充依頼で `baba` キーが異なる（棚卸し: `baba`、補充依頼/移動記録: `baba_main`）。

#### 5.3.2 在庫合計列（補充依頼・移動記録フレーバー一覧共用）

| 列 | 0-indexed | 店舗 |
|----|-----------|------|
| HQ | 224 | 事務所 |
| HR | 225 | 馬場本店 |
| HS | 226 | 中野店 |
| HT | 227 | 馬場2号店 |

#### 5.3.3 移動記録領域

- **領域**: CW〜HL（1-indexed: 101〜220）
- **構造**: 4列 × 最大30ブロック
- **ブロック内オフセット**: `office=0`, `baba=1`, `nakano=2`, `baba_2nd=3`
- **ヘッダー行（行1）**: 左端 = 実施日、右端（+3）= 検品チェックボックス（FALSE/TRUE）
- **データ行（行3〜）**: 移動元は負の値、移動先は正の値で書き込む

### 5.4 棚卸し送信ペイロード

```json
{
  "storeKey": "office | baba | nakano | baba_2nd",
  "sheetName": "1月" | "2月" | ... | "12月",
  "date": "YYYY-MM-DD",
  "items": [
    {
      "rowIndex": 3,
      "brand": "ブランド名",
      "flavorName": "フレーバー名",
      "tupper": { "basic": 0, "reserve": 0 },
      "merch": { "val50": 0, "val100": 0, "val125": 0, "val200": 0, "val250": 0, "val1kg": 0 },
      "flavor": { "val50": 0, "val100": 0, "val125": 0, "val200": 0, "val250": 0, "val1kg": 0, "valOther": 0 }
    }
  ]

```

### 5.5 移動記録送信ペイロード（`submitTransferRecord`）

```json
{
  "monthNum": 4,
  "fromStoreKey": "baba | null（入荷時）",
  "destStoreKey": "nakano | null（廃棄時）",
  "date": "YYYY-MM-DD",
  "autoInspect": true,
  "items": [ { "rowIndex": 3, "qty": 500 } ]
}
```

- `fromStoreKey` が `null` の場合: 移動元列への書き込みを省略（入荷用途）
- `destStoreKey` が `null` の場合: 移動先列への書き込みを省略（廃棄用途）
- `autoInspect: true` の場合: チェックボックスに即 TRUE を書き込む（入荷・廃棄）

## 6. 業務別仕様

### 6.1 棚卸し入力（`InventoryApp.vue`）

#### 画面フロー

```
[トップ: 店舗・月・日付選択] → [店舗確認モーダル]
  → [Step1: タッパー入力] → [Step2: 物販入力] → [Step3: 在庫入力] → [Step4: 確認]
  → [送信確認モーダル] → [submitData] → 完了
```

事務所の場合: Step1（在庫）→ Step2（確認）の2ステップ構成（`currentStep` は 3・4 を使用）。

#### 主要仕様

| 項目 | 内容 |
|------|------|
| 下書き保存 | `localStorage` キー: `inventory_draft_{storeKey}_{month}`。`saveLocally()` を `nextStep` / `prevStep` / `beforeUnmount` 時に呼ぶ |
| 復元確認 | `loadInventoryData()` 内で保存データを検出したとき `requestConfirm` で確認 |
| トップ復帰時確認 | `step → 0` 時、または `returnToPortal()` 時に「未送信・ブラウザ保存」のメッセージで確認 |
| 前月消費マイナス確認 | Step4 遷移時に `checkNegativeConsumption` を非同期実行し、Step4 画面内のバナーに表示（送信は可能） |
| 送信確認 | `InventoryApp` 内の `showSubmitConfirmModal`（Vue カスタムモーダル）で実施 |
| 物販・在庫のグラム入力 | A 列 `brand` 全文（例: `AF (Al Fakher)`）をキーに `src/constants/inventoryPackageRules.js` の静的マスタで表示列を絞り込み。未登録ブランドは従来どおり 50〜1kg 全列＋在庫は「その他」。`MH (Musthave)` は物販パッケージ欄なし・在庫は「その他」のみ |
| タッパー1・2「なし」トグル | Step1 でラベル行右端のスイッチ（既定オン＝数値入力可）。オフ時は入力欄を `disabled` にし、送信・途中保存では該当セルに半角 `-` を書き込む（GAS `buildWriteValues_`）。`getSheetData` はセルが半角 `-` のとき `basicEnabled` / `reserveEnabled` を `false` とし数値は空として返す。数値のみの文字列は従来どおりパース。Step4 は `基:-` / `予:-` で表示 |

Source: [[V-MINT/notes/inventory_package_visibility_plan]]

### 6.2 補充依頼（`RequestApp.vue`）

#### 主要仕様

| 項目 | 内容 |
|------|------|
| 依頼元店舗キー | `baba_main`, `nakano`, `baba_2nd`（事務所は依頼元対象外） |
| データ取得 | `getInventoryData(monthNum)` → `stock: { office, baba_main, nakano, baba_2nd }` |
| 在庫表示 | 依頼元店舗の在庫を右上に大きく表示、他3拠点を小カードで表示（最大在庫拠点をハイライト） |
| 在庫アラート | 500g 未満: 赤・`!` マーク、1000g 未満: 赤 |
| ゼロ非表示 | `requestHideZero` トグルで全拠点 0 のフレーバーを非表示 |
| 出力 | 「内容を確認」→ モーダル内でテキスト生成・コピー（スプレッドシート直接書き込みなし） |

### 6.3 移動記録（`TransferApp.vue`）

#### サブモードと操作フロー

| サブモード | フロー | `autoInspect` | `fromStoreKey` | `destStoreKey` |
|-----------|--------|:---:|---|---|
| 起票（issue） | トップ設定 → Step1A（数量入力）→ Step2A（確認）→ 送信 | false | 選択店舗 | 選択店舗 |
| 検品（inspect） | トップ設定 → 未検品リスト選択 → Step1B（確認・数量修正）→ 完了 | — | — | — |
| 入荷（arrival） | トップ設定（実施店舗・日付）→ Step1A → Step2A → 送信 | true | null | 選択店舗 |
| 廃棄（dispose） | トップ設定（実施店舗・日付）→ Step1A → Step2A → 送信 | true | 選択店舗 | null |

#### 入荷時の新規銘柄追加

- 対象: 入荷トップ設定画面（`transferSubMode === 'arrival'`）
- 入力:
  - `brand`（A列）: 既存ブランドのプルダウン選択
  - プルダウンで `その他` 選択時のみ自由記述欄を表示
  - `flavorName`（B列）: テキスト入力
- 追加API: `addFlavorForArrival({ monthNum, brand, flavorName })`
- 重複判定: A+B（trim後）の完全一致で拒否
- 入力規則:
  - フレーバー名: 各単語の頭文字のみ大文字、単語間は半角スペース（例: `Mexi Cola`）
  - ブランド名（その他入力時）: `大文字略称 + 半角スペース + (正式名称)` 形式。正式名称は各単語の頭文字のみ大文字（例: `AF (Al Fakher)`, `AZR GOLD (Azure Gold Line)`）
  - ルール違反時は警告文を表示し、「この銘柄を追加」を無効化
- 挿入順: ブランド全体・同ブランド内フレーバーともにアルファベット順
- 行初期化:
  - **コピー禁止列**: `C~E, G~J, L~R, T~Y, AB~AE, AG~AM, AO~AT, AW~AZ, BB~BH, BJ~BO, BR~BU, BW~CC, CE~CJ, CM~CN, CW~HL, IG~IL, IO, IP`
  - **0初期化列**: `CO~CV`（コピーではなく数値 `0` を設定）
  - **IP列**: チェックボックス表示を維持し、値は `TRUE`
  - **その他列**: 近傍行の数式をコピー（先頭挿入は直下、末尾挿入は直上）
- 安全性: 行挿入後の初期化失敗時は挿入行を削除してロールバック

#### 起票入力画面の在庫表示

- 起票・廃棄・入荷の Step1A に `getInventoryData` と同一列（HQ〜HT）の在庫を表示
- 移動元（または実施店舗）の在庫を右上に大きく、他3拠点を小カードで表示
- `baba` キー（TransferApp 内）↔ `baba_main` キー（在庫表示）のマッピング: `{ office→office, baba→baba_main, nakano→nakano, baba_2nd→baba_2nd }`

#### 起票ルートのヘッダー表示

- `TransferApp` が `issueFromStore` / `issueDestStore` 変化を watch して `update:issueFromName` / `update:issueToName` を emit
- `App.vue` → `AppHeader` に渡し、Step1A 中のみ右端にコンパクトなルート表示（バッジ＋矢印アイコン）

#### `transferStep` の値と対応画面

| 値 | 画面 |
|----|------|
| `0` | トップ設定画面 |
| `'1a'` | 起票・入荷・廃棄の数量入力 |
| `'2a'` | 起票・入荷・廃棄の内容確認 |
| `'1b'` | 検品（数量確認・チェック）|

### 6.4 在庫量確認（`StockApp.vue`）

| 項目 | 内容 |
|------|------|
| データ取得 | `getStockOverview(monthNum)` |
| 表示内容 | フレーバー別: 総在庫量（g）・前月消費量（g）・全拠点内訳 |
| 性質 | 参照のみ（送信なし） |
| アラート | 総在庫 0 以下の行をハイライト |

## 7. ナビゲーション仕様

### 7.1 戻るボタン・ブラウザバックの統一

`popstate` イベントが発火した際、現在の状態を `history.pushState` で再積みした後に `prevStep()` または `returnToPortal()` を呼ぶことで、**ブラウザバックとフッター「戻る」が同一の挙動**になる。

### 7.2 確認ダイアログが出るタイミング

| 操作 | 確認が出る条件 | メッセージ |
|------|----------------|------------|
| 棚卸し: Step → トップ | Step が 0 になる場合（step1→0, step3→0） | 未送信・ブラウザ保存の旨 |
| 棚卸し: ヘッダー戻る（入力中） | `currentStep > 0` | 同上 |
| 補充依頼: フッター戻る | `currentStep === 1` | 入力破棄・トップ復帰の旨 |
| 移動記録: Step1A/1B → トップ | 常に | 入力内容破棄の旨 |
| 移動記録: Step2A → Step1A | 出ない（データは破棄しない） | — |
| 移動記録: ヘッダー戻る（途中） | 常に | 入力内容破棄の旨 |
| 移動記録: ヘッダー戻る（トップ・入力あり） | 月・店舗等が選択済みの場合 | 入力内容破棄の旨 |
| 移動記録: 送信確認 | 常に | データを記録の可否 |
| 移動記録: 検品数量修正 | 変更があった場合 | 上書き確認 |
| 移動記録: 検品完了 | 常に | 完了確認 |
| 棚卸し: 送信 | 常に | `InventoryApp` 内カスタムモーダル |

### 7.3 フッター（`AppFooter.vue`）の表示条件

```
表示される条件:
  (appMode !== null && currentStep > 0)
  OR
  (appMode === 'transfer' && transferStep !== 0)
  OR
  (appMode === 'stock' && stockHasData)
```

| ボタン | 表示条件 | emit |
|--------|----------|------|
| 戻る | フッター表示時 常に | `prev-step` |
| 次へ | 棚卸し Step 1〜3 | `next-step` |
| 完了する | 棚卸し Step 4 | `submit-inventory` |
| 内容を確認 | 移動記録 Step 1A | `go-to-issue-confirm` |
| データを記録 | 移動記録 Step 2A | `submit-issue` |
| 検品を完了 | 移動記録 Step 1B | `submit-inspection` |

| 右 | Step表示 | 棚卸し `currentStep > 0` |
| 右 | 移動総量（g） | 補充依頼 `currentStep > 0` |
| 右 | 移動ルートバッジ（移動元→移動先） | 移動記録 起票 Step1A 中 |

## 9. セキュリティ・運用上の注意

- `.env` は **リポジトリに含めない**（`.gitignore` 済み）
- `VITE_API_KEY` は GAS 側でも検証を行う（`Code.gs` 内 `validateApiKey_` 等）
- PIN はクライアント側のみの検証のため、端末紛失時は運用でカバーする
- GAS の実行ロックは `LockService.getScriptLock()` で制御（書き込み競合対策）

## 10. 既知の留意点

- **店舗キーの不整合**: `baba`（棚卸し・移動記録）と `baba_main`（補充依頼・在庫合計列）が異なる。フロント・GAS ともにキーのマッピングを維持すること
- **GAS の実行時間制限**: 一度の書き込み量が多い場合（大量行）は GAS の 30 秒制限に抵触する可能性がある
- **移動記録の月内上限**: 1シートあたり最大 30 ブロック（= 月内 30 件）。超過時は GAS がエラーを返す
