---

## 1. システム概要

### 1.1 目的

複数店舗の **棚卸し入力・補充依頼・移動記録・在庫参照** を、スマートフォン／PC ブラウザから一元操作し、**Google スプレッドシート**（GAS バックエンド）と連携する。

### 1.2 アーキテクチャ

```
[ブラウザ: Vue 3 SPA]
        │ HTTPS (fetch)
        ▼
[GAS Web アプリ URL]
        │
        ▼
[Google スプレッドシート / スクリプト処理]
```

- フロントは **静的ホスティング**（例: Cloudflare Pages）想定。
- 認証は **PIN（4桁）** をクライアント側で検証（本番では GAS 側でも API キー検証を推奨）。
- API キー・GAS URL は **環境変数**（`VITE_*`）で注入。

### 1.3 技術スタック

| 区分 | 技術 |
|------|------|
| フレームワーク | Vue 3（Options API） |
| ビルド | Vite 5 |
| スタイル | Tailwind CSS 3 |
| HTTP | `fetch`（`api.js` に集約） |

## 4. アプリケーション状態モデル

### 4.1 `appMode`（業務モード）

| 値 | 意味 |
|----|------|
| `null` | ポータル |
| `inventory` | 棚卸し入力 |
| `request` | 補充依頼 |
| `transfer` | 移動記録 |
| `stock` | 在庫量確認 |

### 4.2 履歴（ブラウザ）

- `history.pushState` / `popstate` で `appMode`, `currentStep`, `transferStep` を保持。
- 戻る操作で状態を復元。

### 4.3 グローバル UI

- **ローディング**: `LoadingOverlay`（子から `loading` / `loadingMessage` を emit）。
- **ブランド絞り込み**: `BrandFilterSheet`（モードごとに `brands` ソースを切替）。

## 5. バックエンド API（`api.js`）

### 5.1 通信方式

| メソッド | 用途 | 備考 |
|----------|------|------|
| GET | `gasGet(action, params)` | クエリ `action`, `apiKey`, 各パラメータ |
| POST | `gasPost(action, payload)` | **Body は `Content-Type: text/plain`**（GAS CORS 制約回避） |
| | | JSON 文字列: `{ action, apiKey, payload }` |

エラー時: レスポンス JSON の `error.message` を `throw`（GET/POST 共通）。

### 5.2 アクション一覧（フロント定義）

| 関数 | GAS `action`（GET/POST） | 主な用途 |
|------|--------------------------|----------|
| `getSheetData(storeKey, sheetName)` | `getSheetData` | **棚卸し** 行データ取得（`sheetName` は月ラベル例: `4月`） |
| `submitData(payload)` | `submitData`（POST のトップ `action`） | **棚卸し** 送信 |
| `getInventoryData(monthNum)` | `getInventoryData` | **補充依頼** 在庫一覧取得（数値月） |
| `getFlavorListForTransfer(monthNum)` | `getFlavorListForTransfer` | 移動記録 |
| `getPendingTransferRecords(...)` | `getPendingTransferRecords` | 移動記録 |
| `getTransferRecordDetail(...)` | `getTransferRecordDetail` | 移動記録 |
| `submitTransferRecord(payload)` | `submitTransferRecord` | 移動記録 |
| `completeInspection(payload)` | `completeInspection` | 移動記録 |
| `getStockOverview(monthNum)` | `getStockOverview` | 在庫量確認 |

> **重要（棚卸し）:** データ取得は **`getSheetData`** を使用する。`getInventoryData` は補充依頼用であり、行番号（`rowIndex`）が棚卸し送信と整合しない。誤用すると GAS 側でシート書き込みエラーとなる。

### 5.3 棚卸し送信ペイロード（`submitData` の `payload` 内）

GAS が期待する形（実装確認済みの契約）:

```json
{
  "storeKey": "office | baba | nakano | baba_2nd",
  "sheetName": "1月" ～ "12月",
  "date": "YYYY-MM-DD",
  "items": [ /* 各行: rowIndex, brand, flavorName, tupper, merch, flavor 等 */ ]
}
```

- `items` は `getSheetData` 応答をベースに、入力 UI 用オブジェクトへ正規化した配列。

### 5.4 棚卸し送信レスポンス（想定）

- `success: true` 時、`negativeConsumptionItems`（文字列配列）があれば **前月消費マイナス警告** 用に表示。

## 6. 業務別仕様

### 6.1 棚卸し入力（`InventoryApp.vue`）

| 項目 | 内容 |
|------|------|
| 店舗 | `App.vue` の `stores`（`office`, `baba`, `nakano`, `baba_2nd`） |
| 月 | `1月`～`12月` 文字列 |
| ステップ | 通常: 1 タッパー → 2 物販 → 3 在庫 → 4 確認。**事務所**は 3→4 のみ（表示ステップは 2） |
| データ取得 | `getSheetData(storeKey, month)` → `items` をマッピング |
| 送信 | フッター「完了する」→ `App.vue` → `ref.submitInventory()` → `submitData` |
| 下書き | `localStorage` キー `inventory_draft_{storeKey}_{month}`。`saveLocally()` は **InventoryApp** 内。`nextStep` 時および `beforeUnmount` で保存 |

### 6.2 補充依頼（`RequestApp.vue`）

| 項目 | 内容 |
|------|------|
| 店舗キー | UI 上 `baba_main`, `nakano`, `baba_2nd`（**棚卸しの `baba` とは別キー**） |
| データ | `getInventoryData`（月はコンポーネント内の値形式） |
| 備考 | GAS 返却の `stock` オブジェクトキーと整合させる必要あり |

### 6.3 移動記録（`TransferApp.vue`）

| 項目 | 内容 |
|------|------|
| サブモード | 起票 / 検品 / 入荷 / 廃棄 |
| ステップ | `transferStep`（数値 0 および文字列 `'1a'`, `'2a'`, `'1b'` 等） |
| フッター | `AppFooter` から emit → `App.vue` が `ref.transferApp` に委譲 |

### 6.4 在庫量確認（`StockApp.vue`）

| 項目 | 内容 |
|------|------|
| 操作 | 月選択後 `getStockOverview` |
| 性質 | **参照のみ**（送信なし） |

## 7. UI / UX 仕様（抜粋）

### 7.1 固定フッター（`AppFooter.vue`）

- `position: fixed; bottom: 0`。
- 棚卸し Step 1～4 では、メイン領域に **十分な `padding-bottom`**（＋ `safe-area-inset-bottom`）を確保し、最終行入力と重ならないようにする。

### 7.2 PIN 認証（`PinAuth.vue`）

- 成功時 `@authenticated` → `App.vue` の `authenticated = true`。
- PIN 値の保持場所はコンポーネント実装に依存（本書ではコード参照）。

---

## 8. セキュリティ・運用上の注意

- `.env` は **リポジトリに含めない**（`.gitignore` 済み）。
- API キーは **GAS 側でも検証**することを推奨。
- PIN は端末内のみの場合、**端末紛失リスク**を運用でカバーする。

---

## 9. 既知の留意点・技術負債（参考）

- `App.vue` に **棚卸し開始用のレガシー処理**（`startInventory` / `confirmStartInventory` / `showInventoryStartModal`）が残存している可能性があり、実際の開始フローは **`InventoryApp` 内モーダル**が主。コンソール警告の原因になり得る。
- `AppHeader` の `requestCurrentStoreName` 等、**未配線 props** による Vue warn が出る場合あり。
- 補充依頼と棚卸しで **店舗キー文字列が異なる**（`baba` vs `baba_main`）。GAS・マスタとの整合を維持すること。
