## 1. プロジェクトの背景と出発点

### 1.1 解決したかった課題

複数の水タバコ（シーシャ）店舗では、毎月の棚卸し・在庫管理・店舗間の商品移動が手作業で行われていた。主な問題点は以下のとおりだった。

- 月末棚卸しの集計が **スプレッドシートへの手入力**で、転記ミスや抜け漏れが発生しやすかった
- 在庫が不足したときの補充依頼が口頭・メッセージアプリ頼りで、記録が残らなかった
- 店舗間の商品移動（起票・検品）に**統一フォーマットがなく**、トラブルの原因になっていた
- 担当者がスプレッドシートを**直接触らなくて済む UI** がなかった

### 1.2 最初の構成（フェーズ 1）

最初のプロトタイプは **GAS（Google Apps Script）で HTML を配信する**構成だった。

```
[GAS HtmlService（Play CDN TailwindCSS）]
    ↕ google.script.run（非同期）
[GAS スクリプト処理]
    ↕
[Google スプレッドシート × 4ファイル（店舗別）]
```

- フロントとバックエンドが同一の GAS プロジェクトに同居
- 4 店舗分のスプレッドシートがそれぞれ独立したファイル
- `google.script.run` 経由の非同期呼び出し

この構成でまず棚卸し入力・補充依頼を実装し、現場でのテスト・フィードバック収集を開始した。

## 2. フェーズ 2: バックエンドの統合と移動記録モードの追加

### 2.1 スプレッドシートの1ファイル統合

4 店舗分を別々のファイルで管理していた構成を、**月ごとのシートを持つ 1 ファイル**に統合した。

- **効果**: GAS がファイルを開く処理が 4 回 → 1 回になり、最初のデータ読み込みが数秒から大幅に短縮された
- **副次効果**: 全店舗の在庫をひとつのシートで横断参照できるようになり、補充依頼・在庫量確認などクロス店舗の機能実装が容易になった

### 2.2 移動記録モードの追加

バックエンド統合に合わせて「移動記録」モードを新規実装した。

- **起票**: 移動元・移動先・数量を記録してスプレッドシートに書き込む
- **検品**: 受け取り側が数量を確認・修正し、完了チェックを付ける
- **廃棄**: 廃棄した数量をマイナス計上として記録

スプレッドシートの列設計:

```
CW〜HL列（101〜220列）= 移動記録専用領域
4列1ブロック × 最大30ブロック（月内30件まで）
ブロック内: 事務所列 / 馬場列 / 中野列 / 2号店列
行1: 実施日（左端）・検品チェックボックス（右端）
行3〜: フレーバーごとの数量（移動元: 負の値、移動先: 正の値）
```

LockService による排他制御を実装し、複数ユーザーの同時書き込みによるデータ破損を防止した。

## 3. フェーズ 3: フロントエンドの Vite/Vue3 移行

### 3.1 移行の理由

GAS HtmlService での HTML 配信には次の限界があった。

- HTML が肥大化するにつれ **初期表示が遅く**なり始めていた
- Play CDN（Tailwind の CDN 版）では**ブラウザ描画中のカクつき**（未変換クラスの一時的な表示）が発生していた
- コード量が増え `Code.gs` と HTML が混在し、**開発・デバッグが困難**になっていた

### 3.2 新アーキテクチャ

```
[Vue 3 SPA（Vite ビルド → Cloudflare Pages で静的配信）]
        │ HTTPS fetch（Content-Type: text/plain）
        ▼
[GAS Web アプリ URL（/exec）= 外部から叩ける JSON API]
        │
        ▼
[Google スプレッドシート（1ファイル・月別シート）]
```

GAS 側は「HTML を配信する」役割を完全に廃止し、**データの読み書きのみを行う API サーバー**として再構成した。

### 3.3 技術的な決断

| 決断 | 選択した方向 | 理由 |
|------|-------------|------|
| フレームワーク | Vue 3（Options API） | 既存コードが Vue ライクな記述をしており、学習コストが最小 |
| ビルドツール | Vite 5 | HMR が速く、設定がシンプル |
| スタイル | Tailwind CSS 3（ビルド時に静的化） | 既存クラスを流用でき、Play CDN のカクつき問題も解消 |
| API 通信 | fetch（`Content-Type: text/plain`） | GAS が `application/json` の CORS プリフライトをブロックするための回避策 |
| ホスティング | Cloudflare Pages（無料枠） | 静的ファイルを CDN 配信でき、デプロイも push だけで完結 |

### 3.4 コンポーネント分割

モノリシックだった `App.vue`（約2,000行超）を業務ごとに分割した。

```
App.vue（ルート・状態管理）
├── common/
│   ├── PinAuth.vue        PIN 認証
│   ├── AppHeader.vue      ヘッダー
│   ├── AppFooter.vue      フッター（次へ・戻る・送信）
│   ├── LoadingOverlay.vue ローディング
│   └── BrandFilterSheet.vue ブランド絞り込み
└── apps/
    ├── InventoryApp.vue   棚卸し入力
    ├── RequestApp.vue     補充依頼
    ├── TransferApp.vue    移動記録
    └── StockApp.vue       在庫量確認
```

**分割後の効果:**
- App.vue が各モードの「状態管理・画面切替・親イベントの仲介」に専念できるようになった
- 業務ロジックが各 App コンポーネントに閉じており、他業務に影響せず改修できる

## 4. リリース後の継続改善

### 4.1 棚卸し入力の完成度向上

| 改善 | 内容 |
|------|------|
| 下書き保存 | `localStorage` に step 移動のたびに保存。次回同じ店舗・月で開始したとき復元を確認 |
| 前月消費マイナスチェック | Step4 遷移時にバックエンドへドライランを実行（`checkNegativeConsumption`）し、Step4 画面内のバナーで警告表示。送信自体は可能 |
| 送信確認モーダル | ネイティブ `confirm()` からカスタムモーダルに移行し、全環境で確実に表示されるようにした |

### 4.2 補充依頼の在庫表示強化

- フレーバーカードに **依頼元店舗の現在在庫**を大きく表示（500g 未満は赤・警告マーク）
- 他3拠点の在庫をコンパクトなカードで横並び表示（最大在庫拠点をグリーンハイライト）
- **「在庫ゼロ非表示」** トグルで全拠点 0 のフレーバーを一括非表示

### 4.3 移動記録の機能拡充

**在庫量の表示（起票・廃棄・入荷の入力画面）:**
- 補充依頼と同一のデータ（HQ〜HT列）を `getFlavorListForTransfer` のレスポンスに追加
- 起票・廃棄・入荷の Step1A（数量入力画面）で、移動元/実施店舗の在庫を各フレーバーカード内に表示
- 他3拠点との比較カードも同時表示し、移動量の判断材料にできる

**起票ルートのヘッダー表示:**
- 数量入力画面でスクロールしても移動元→先が常に見えるよう、AppHeader 内にコンパクトなバッジ形式で表示
- TransferApp → App.vue → AppHeader とデータを emit で伝播する設計

**入荷の全店舗対応:**
- 従来は事務所のみだった入荷記録を、全拠点で記録できるよう拡張
- GAS 側は `destStoreKey` に選択店舗を渡すだけで対応（既存の列振り分けロジックを流用）

### 4.4 ナビゲーション・UX の全面改善

**ネイティブ `confirm()` の廃止（`ConfirmDialog.vue`）:**

モバイル Chrome・PWA・一部 WebView 環境では `window.confirm()` がダイアログを表示せず即 `false` を返すことがあり、確認を出しているつもりが出ていないという問題が発生していた。

全ての確認ダイアログをアプリ内 Vue コンポーネントに統一した。

```
App.vue が provide() で requestConfirm を提供
    ↓
TransferApp / InventoryApp が inject して使用
    ↓
await this.requestConfirm('メッセージ') が true/false を返す Promise
```

**ブラウザバックと戻るボタンの統一:**

従来は `popstate` で `event.state` を直接反映していたため、確認ダイアログなしに画面が遷移していた。

```javascript
handlePopState() {
  // 現在の状態を再 push → history 上の位置をバック前に戻す
  history.pushState({ appMode, currentStep, transferStep }, '')
  // 通常の戻るロジック（確認ダイアログ込み）を呼ぶ
  if (footerBackVisible) this.prevStep()
  else if (appMode !== null) this.returnToPortal()
}
```

これにより、ブラウザバックでも確認ダイアログが表示されるようになった。

**各モードの戻る挙動整理:**

| 状態 | フッター戻る | ヘッダー戻る |
|------|------------|------------|
| 棚卸し Step→0 | 確認（保存通知）→ トップ | 確認（保存通知）→ ポータル |
| 棚卸し Step→Step | 確認なし | — |
| 移動記録 Step1A/1B | 確認（破棄警告）→ 移動記録トップ | 確認（破棄警告）→ 移動記録トップ |
| 移動記録 Step2A | 確認なし → Step1A | — |
| 移動記録トップ（入力あり）| — | 確認（破棄警告）→ ポータル |
| 補充依頼 Step1 | 確認（破棄警告）→ 補充依頼トップ | — |

## 6. 技術的に難しかった点と解決策

### 6.1 GAS の CORS 制約

GAS の Web アプリは `Content-Type: application/json` の POST リクエストに対してプリフライトを拒否する。

**解決策**: `Content-Type: text/plain` で送信し、Body を JSON 文字列にする。GAS 側で `JSON.parse(e.postData.contents)` で受け取る。

### 6.2 ネイティブダイアログのブロック問題

モバイル Chrome・一部 WebView 環境での `window.confirm()` のサイレント無効化。

**解決策**: Vue の `provide` / `inject` を使ったアプリ全体の共通確認ダイアログ（`ConfirmDialog.vue`）を実装し、ネイティブダイアログを全廃した。

### 6.3 ブラウザバックと確認ダイアログの共存

`popstate` が発火した時点でブラウザはすでに history を 1 つ戻らせており、確認後にキャンセルしても history 上の位置が変わらないという問題。

**解決策**: popstate 発火時に `history.pushState` で現在状態を再積みし直してから確認ダイアログを表示する「インターセプト方式」を採用。

### 6.4 移動記録の排他制御

複数ユーザーが同時に起票したとき、同一ブロックに二重書きされるリスク。

**解決策**: GAS の `LockService.getScriptLock().tryLock(10000)` で書き込みをシリアライズし、空きブロックの確定と書き込みをアトミックに実行。

## 8. 移行ロードマップ上の位置づけ

本プロジェクトは `migration_plan.md` に示したロードマップの **フェーズ 3** に相当する。

```
フェーズ 1: GAS HtmlService プロトタイプ ✅ 完了
フェーズ 2: スプレッドシート統合 + 移動記録追加 ✅ 完了
フェーズ 3: Vue3/Vite フロントエンド + GAS JSON API ✅ 現在ここ（リリース済み）
フェーズ 4: RDB 移行（将来的な視野）
```

フェーズ 3 の段階で「無料 Cloudflare Pages + GAS API」の組み合わせが十分快適であることが確認できており、**現行構成（フェーズ 3）を長期的な運用形態として維持する**選択肢が有力。

フェーズ 4（データベース完全移行）への移行判断は、以下のトリガーが発生したときに再検討する:

- 月内の移動記録件数が 30 件を超えるケースが常態化した場合
- 同時多ユーザー操作によるデータ競合が実際に問題化した場合
- 高度な集計・検索クエリのニーズが生じた場合
